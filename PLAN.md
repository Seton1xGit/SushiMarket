# SushiMarket TR — WhatsApp Order Notifications

Проект: автоматические WhatsApp-уведомления клиентам по жизненному циклу заказа на турецкой инсталляции SushiMarket / DataFood (`office-tr.sushi-market.com`, админка `call-center-tr.sushi-market.com`).

> Документ — спецификация требований и план (output `/sc:brainstorm`).
> Содержит: цели, функциональные/нефункциональные требования, user stories,
> предложенный стек, найденные методы API, зафиксированные решения, риски.
> Архитектурная схема и реализация — отдельным шагом (`/sc:design`, `/sc:implement`).

---

## 1. Цели

- При появлении нового заказа-доставки на сайте — забрать данные о заказе, клиенте и составе, отправить клиенту приветственное сообщение в WhatsApp.
- Поймать переход статуса «готовится → доставляется» — отправить клиенту сообщение «заказ в пути + ориентировочное время доставки» (если зона настроена).
- Через 2 часа после `completed` отправить клиенту запрос обратной связи / NPS — но только если расчётное время отправки не позже 22:00 локального времени.
- Решение должно быть с минимальной операционной стоимостью, поддерживаемым и устойчивым. Платформенные расходы (WhatsApp Cloud API, ~$5–10/мес при пиковой нагрузке) — приемлемы.

## 2. User Stories

| # | Роль   | Действие                                                                 | Ценность                       |
|---|--------|--------------------------------------------------------------------------|--------------------------------|
| 1 | Клиент | Получает в WA подтверждение заказа со списком позиций сразу после оформления | Чувствует, что заказ принят    |
| 2 | Клиент | Получает в WA уведомление «курьер выехал, ~XX минут» в момент отправки    | Знает, когда ждать             |
| 3 | Клиент | Через 2 часа после получения (если не позже 22:00) получает короткий NPS-опрос | Может оставить отзыв          |
| 4 | Бизнес | Не пишет лишних сообщений в неурочное время и не дублирует одно событие   | Не получает жалоб «спам ночью» |
| 5 | Бизнес | Видит лог отправок и ошибок, может вручную переотправить                  | Контроль качества              |

### Acceptance criteria

- AC-1: При создании заказа за ≤ 5 минут клиенту приходит WA «Здравствуйте, ваш заказ #N принят…» с составом и суммой.
- AC-2: Когда статус заказа становится `courier_delivers_an_order`, клиенту в течение ≤ 5 минут приходит WA «Курьер выехал, ориентировочное время доставки ~XX минут».
- AC-3: Через 2 часа после статуса `completed` клиенту приходит WA с NPS-вопросом, **если расчётное время отправки ≤ 22:00 локального времени Europe/Istanbul**. Если позже — NPS не отправляется вообще (см. §10.4).
- AC-4: Каждое из трёх сообщений по одному заказу отправляется ровно один раз (идемпотентность).
- AC-5: Если у клиента нет валидного телефона — лог + эскалация админу в Telegram, без отправки клиенту (см. §10.6).
- AC-6: Все 3 сообщения отправляются **только для заказов с типом «доставка»**. Для самовывоза — никаких сообщений (см. §10.3).
- AC-7: Сообщение «курьер в пути» отправляется только если для зоны заказа настроено `customDeliveryTime`. Иначе — не отправляется + лог (см. §10.5).

## 4. Найденные методы (что нам пригодится)

Используем **два независимых источника**:
- **Триггер** — admin-API call-center: `https://sushimarket-tr.datafood.tech/call-center/api/...` (Bearer админа). Из него берём ТОЛЬКО `(order_id, ClientPhone)` и переходы статусов.
- **Данные о заказе** — публичный API DataFood: `https://office-tr.sushi-market.com/api/app/{version}/{method}` (Bearer интегратора). Все детали — через него.

Такое разделение минимизирует зависимость от deprecated-эндпоинта call-center'а: при его поломке встаёт только триггер, бизнес-логика не страдает.

### 4.0 Admin-API call-center (триггер)

| Метод | URL | Назначение |
|-------|-----|------------|
| GET   | `https://sushimarket-tr.datafood.tech/call-center/api/orders__deprecated?status=activ&type_select_cities=include` | Листинг всех активных заказов. Возвращает массив объектов с `id`, `number`, `OrderDate`, `ClientName`, `ClientPhone`, `Address`, `OrderAmount`, `Status`, `order_type`, `city_id`, `terminal`, `Courier`, `Seller`, `paid` и др. |

Заголовки запроса (обязательны):
```
Authorization: Bearer <ADMIN_TOKEN>      # формат "70227|Hci2Kf..."
Origin:  https://call-center-tr.sushi-market.com
Referer: https://call-center-tr.sushi-market.com/
Accept:  application/json, text/javascript, */*; q=0.01
```

Параметр `_={Date.now()}` — cache-buster, безвреден.

⚠ **Endpoint помечен `__deprecated`** — vendor может выпилить. См. §13.

### 4.1 Публичный API DataFood

База: `https://office-tr.sushi-market.com/api/app/{version}/{method}` + `Authorization: Bearer <PUBLIC_TOKEN>`.

### 4.2 Получение данных заказа

| Метод | URL | Ключевые поля ответа | Применение |
|-------|-----|----------------------|------------|
| GET   | `/v1/client/orders/get_for_phone?order_id&order_phone` | products, delivery, shop, payments, status, created_at, order_date | **Основной путь.** Телефон уже есть из листинга админки (§4.0). |
| GET   | `/v1/client/orders/get?order_id&client_id`              | то же + расширенно                                                | Резерв, если клиента идентифицируем по `client_id`. В текущем дизайне не используется. |
| GET   | `/v1/client/get/{id}`                                   | phone, full_name, email, city, loyalty                            | Резерв. В текущем дизайне не используется. |

⚠ **TBD:** В ответе `get_for_phone` нужно подтвердить, есть ли поле `zone_id` (или эквивалент) для запроса `customDeliveryTime`. Если нет — потребуется маппинг `city_id → zone` или другой источник зоны.

### 4.3 Статусы и опрос (резерв)

| Метод | URL | Назначение |
|-------|-----|------------|
| POST  | `/v5/order/status` (body: `{"orders":[id1,id2,…]}`) | Пакетный опрос статусов нескольких заказов |
| POST  | `/v5/order/status_for_phone`                        | Статус одного заказа по телефону            |
| GET   | `/v1/content/statusList`                            | Полный enum статусов с человекочитаемыми именами (для маппинга) |

Полный жизненный цикл (по доке v5):
```
new → waiting_for_filling → waiting_for_shop_confirmation
    → confirmed_by_shop → cooking
    ├─ delivery: → waiting_for_courier → courier_delivers_an_order → completed
    └─ pickup:   → cooked → completed
                                    canceled (любой шаг)
```

«Готовится → доставляется» = `cooking → courier_delivers_an_order` (delivery).
Для самовывоза аналог — `cooking → cooked`, но «в пути» там не уместно.

### 4.4 Время доставки / приготовления

| Метод | URL | Возвращает |
|-------|-----|------------|
| GET   | `/v1/get_custom_delivery_time/{zone}` | `{"minutes": N}` — кастомное время доставки по зоне |
| GET   | `/v1/get_custom_cooking_time/{shop}`  | `{"minutes": N}` — кастомное время готовки по магазину |
| GET   | (внешний RU) `https://call-center.sushi-market.com/api/courier/get-location?order_id` | GPS курьера. Только RU-домен в доке; для TR — **проверить аналог**, в текущем scope не используется. |

ETA для AC-2: вызываем `get_custom_delivery_time/{zone}`. Если ответа нет / `minutes` не настроены — сообщение «в пути» **не отправляется** (см. §10.5).

### 4.5 Чего нет в публичном API

- **Webhooks / push** — DataFood их не поддерживает. Все интеграции работают по polling.
- **Лист новых заказов магазина** в публичной `appClient/*` доке не описан. Закрыто через admin-API call-center (§4.0).

## 5. Триггер «новый заказ»

Принято: **polling листинга admin-API call-center** (§4.0) каждые 15 сек.

Схема:
```
каждые 15 сек:
  GET /call-center/api/orders__deprecated?status=activ&type_select_cities=include
  → массив активных заказов
  для каждой строки берём ТОЛЬКО:
      order_id    = data[i].id
      phone       = data[i].ClientPhone
      status      = data[i].Status        # для diff
      order_type  = data[i].order_type    # 1 = delivery, иначе pickup (уточнить)
  diff с предыдущим снимком из БД:
      есть order_id, которого не было  → событие "new_order"
      изменился status у известного    → событие "status_changed"
      исчез order_id из активных       → событие "order_closed"  (значит status стал completed/canceled — на этом моменте срабатывает NPS-планировщик)
для каждого события new_order и status_changed (если нужно):
  GET https://office-tr.sushi-market.com/api/app/v1/client/orders/get_for_phone?order_id=…&order_phone=…
  → полные данные → форматирование → отправка WA
```

Резервный канал на случай, если call-center лёг:
- `POST https://office-tr.sushi-market.com/api/app/v5/order/status` с массивом `id` известных активных заказов — даст хотя бы переходы статусов по уже зафиксированным заказам (новые в это время не получим).

## 6. WhatsApp — провайдер и стоимость (Турция)

Рекомендация: **WhatsApp Cloud API (Meta, официальный)**.

- **Utility-категория** (наш кейс — транзакционные уведомления о заказе): в Турции по обновлению с 1 апреля 2026 — ~$0.0008 за сообщение (одна из самых низких ставок мира). Шаблоны должны быть согласованы заранее.
- **Service-сообщения** (ответ клиента в течение 24 ч после его сообщения) — бесплатны.
- **FEP-окно**: если клиент пришёл с click-to-WhatsApp рекламы — 72 ч бесплатной переписки.
- **Бесплатной квоты на массовые шаблоны нет** (старый «1000 free conversations / month» Meta убрала летом 2025).

Расчёт стоимости (пик 150 заказов/день, см. §10.7):
- ~50% delivery × до 3 шаблонов на заказ = ~225 utility-сообщений/день в верхней оценке.
- 225 × 30 × $0.0008 ≈ **$5.4/мес** на пиковом сценарии.
- Реальная цифра ниже: NPS отсекается окном до 22:00 (§10.4), «в пути» отсекается зонами (§10.5).

Альтернативы:
- **whatsapp-web.js / Baileys** (неофициальные на сессии телефона) — бесплатно, но риск блокировки номера и нарушение ToS Meta. **Не рекомендуется для бизнеса.**
- Twilio, Wati, 360dialog, Whapi.cloud — переплата над Cloud API, не нужны.

## 7. Стек

**Зафиксировано:** n8n Community Edition на существующем хостинге пользователя — `https://n8n.thefreedom.pro` (см. §10.10).

Сравнение с альтернативами (для контекста / при возможной миграции):

| Критерий                          | Чистый код (Node.js/Python) | n8n self-hosted **(выбран)**   | Make (Integromat)     |
|-----------------------------------|------------------------------|--------------------------------|-----------------------|
| Стоимость (наша нагрузка)        | $0 + хостинг                 | $0 (используем существующий)   | Free 1000 ops/мес — выгорит за полдня при polling 15 c |
| Время до MVP                     | 2–4 дня                     | 0.5–1 день                     | 0.5 дня              |
| Отложенные задания (NPS через 2ч)| нужен job-queue (BullMQ)     | встроенный Wait-node           | Sleep-модуль жжёт ops |
| Polling статусов                 | cron + код                  | Schedule trigger (sec)         | Schedule trigger (≥15 мин на free) |
| Кастомная логика                 | максимум                    | Function-node (JS)             | ограниченно           |
| Поддержка не-разработчиком       | нет                         | да (UI)                        | да (UI)               |
| Lock-in / экспорт                | нет                         | workflow.json — переносим      | сценарии не экспортируются |

Причины выбора:
1. Polling раз в 15 с возможен только в self-hosted n8n; Make убил бы лимит, чистый код требует отдельной инфры.
2. Хостинг уже есть и оплачен пользователем.
3. Workflow → JSON в git → передача заказчику без переписывания кода.

## 8. Высокоуровневый план (без конкретных API-контрактов)

1. **Окружение и доступы**
   - Подключить тестовый Meta WhatsApp Business Account (см. §10.8). Сохранить `META_WA_TOKEN`, `META_PHONE_NUMBER_ID`, `META_VERIFY_TOKEN`.
   - Создать и подать на ревью 3 шаблона на русском: `order_received`, `order_in_transit`, `feedback_request`. Тон/эмодзи согласовываем при подготовке текстов.
   - В существующем n8n (`https://n8n.thefreedom.pro`) сохранить креденшалы: `DATAFOOD_PUBLIC_TOKEN`, `CALLCENTER_ADMIN_TOKEN`, `META_*`, `TELEGRAM_*`.
   - Создать Telegram-бота для алертов и канал админа.
2. **Workflow A — детектор изменений в листинге активных заказов**:
   - Schedule trigger каждые **15 сек** → GET листинг call-center (§4.0) → diff с предыдущим снимком → события `new_order` / `status_changed` / `order_closed`.
3. **Workflow B — приветствие** (AC-1, AC-6): подписан на `new_order`. Если `order_type ≠ delivery` → пропускает. Иначе вызывает `get_for_phone`, форматирует состав → отправка WA `order_received`.
4. **Workflow C — «в пути»** (AC-2, AC-7): подписан на `status_changed`. Срабатывает только при `→ courier_delivers_an_order`. Берёт зону из деталей заказа → `GET get_custom_delivery_time/{zone}`. Если `minutes` есть → отправка WA `order_in_transit`. Иначе → лог + флаг `transit_skipped`.
5. **Workflow D — NPS** (AC-3): подписан на `status_changed → completed`. Считает `target_time = completed_at + 2h`. Если `target_time > 22:00 Europe/Istanbul` того же дня → флаг `feedback_skipped`, сообщение **не отправляется**. Иначе — Wait-node до `target_time`, потом отправка WA `feedback_request`.
6. **Хранение состояния** (для идемпотентности и таймеров):
   - Лёгкая БД (SQLite через volume или Postgres, что есть в существующем n8n).
   - Таблица `order_events`:
     - `order_id` (PK)
     - `client_phone`
     - `order_type`
     - `last_status`
     - `last_seen_at`
     - `welcome_sent_at` (NULL если не отправлено)
     - `transit_sent_at`
     - `transit_skipped` (bool — зона не настроена)
     - `feedback_scheduled_at`
     - `feedback_sent_at`
     - `feedback_skipped` (bool — окно после 22:00)
     - `escalated_no_phone_at` (для AC-5)
7. **Логирование и алерты**: ошибки HTTP / 401 / отсутствие телефона / зона не настроена → Telegram-канал админу.
8. **Тест-чеклист**:
   - Заказ-доставка, зона настроена, completed в 19:30: дошли все 3 сообщения; NPS пришёл в 21:30.
   - Заказ-доставка, зона **не настроена**, completed в 19:30: пришло AC-1 и AC-3 (в 21:30), AC-2 пропущен (флаг `transit_skipped`).
   - Заказ-самовывоз: **ни одного сообщения** клиенту (AC-6).
   - Отменённый заказ: после `canceled` ничего не шлём, NPS не планируется.
   - Поздний заказ, completed в 21:30 → target = 23:30 > 22:00 → NPS **не отправляется** (флаг `feedback_skipped`).
   - Заказ без телефона: эскалация в Telegram, клиенту ничего.
   - Дубль-polling после рестарта n8n: каждое сообщение шлётся ровно один раз (по флагам `*_sent_at`).
   - Симуляция падения call-center API (404/500): алерт, новые заказы не теряются — догоняются на восстановлении.

## 8.1 Safety: dry-run mode и правила активации

**Правило (зафиксировано пользователем 2026-04-28):** Никаких тестов на реальных клиентах. Никаких активных workflow в n8n до полной готовности бэка.

Обязательные условия активации (`active: true`):
1. Все 5 WF реализованы и проверены на синтетических order_id.
2. Тест-чеклист §8.8 пройден полностью.
3. Пользователь явно подтвердил активацию.

Во время разработки:
- Все WF держим `active: false` (создаются по умолчанию неактивными; проверяется `n8n-cli workflow list`).
- Manual run (`workflow execute`) допускается **только** на WF, которые не отправляют реальных сообщений клиентам или используют dry-run mode.
- WhatsApp Cloud API в коде нод **по умолчанию dry-run**: вместо HTTP к Meta — INSERT в `notification_log` с `success=false, error_text='dry_run', payload=<сообщение>`. Реальная отправка включается единым флажком (`META_DRY_RUN=true|false` в env, или ENV-condition в IF-ноде перед каждым WA Send).
- Telegram-алертер (WF-E) идёт в личный chat_id админа — это разрешено и не затрагивает клиентов.

**Если автоматизация предлагает активацию — она обязана сначала спросить разрешение пользователя.** Это override любых "auto mode" директив.

## 9. Нефункциональные требования

- Latency: AC-1, AC-2 — детектируется и доставляется ≤ 5 мин от события.
- Reliability: ретраи (3 попытки с экспонентой) при ошибках Meta API и DataFood.
- Idempotency: на каждое сообщение — флаг в БД, повторный запуск не дублирует.
- Privacy: телефон клиента и состав заказа не выходят за пределы проекта; логи без PII или с маскировкой.
- Compliance: все сообщения через approved utility-templates Meta. Никаких рассылок без согласия.
- Time-zone: вся логика времени на `Europe/Istanbul`.

| 6 | Нет валидного телефона у клиента      | Тихо логируем, ничего не отправляем + **эскалация админу** в Telegram (отдельный канал). |

### Производные требования из решений

- В БД `order_events` добавить поле `feedback_skipped` (bool) — для случаев §4 (после 22:00).
- В БД `order_events` добавить поле `transit_skipped` (bool) — для случаев §5 (зона не настроена).
- При импорте workflow в `n8n.thefreedom.pro` — проверить версию и наличие WhatsApp Business Cloud node (нужно ≥ 1.36). Если ниже — апгрейд или собрать запрос через generic HTTP Request node.
- Telegram-канал админа должен быть создан до запуска (нужен `TELEGRAM_BOT_TOKEN` и `ADMIN_CHAT_ID` в env).

- Интеграция с CRM / коллтрекинг / внешние BI.
- Двусторонний чат с курьером.

## 13. Риски и митигация

| Риск | Вероятность | Импакт | Митигация |
|------|-------------|--------|-----------|
| Endpoint `/call-center/api/orders__deprecated` будет выпилен vendor'ом | средняя | высокий — встаёт триггер «новый заказ» | (a) мониторинг 401/404/500, алерт админу за 5 мин; (b) резервный канал — `POST /v5/order/status` по уже известным заказам, не дублирует, но удерживает переходы статусов; (c) периодически (раз в 2 нед) проверять Network в браузере — не появился ли «новый» эндпоинт без `__deprecated` |
| Истечение Bearer-токена админ-API | низкая | средний | алерт на 401; либо запросить у vendor долгоживущий personal-token через ЛК админа |
| Истечение публичного Bearer-токена | низкая | высокий — встаёт всё | алерт на 401; продление через владельца интеграции |
| Meta WhatsApp лимит 1000/день на новом WABA | средняя на старте | средний | при запуске — мониторинг отправок; быстрый переход на Tier-2 после набора активных диалогов |
| Шаблон не одобрен Meta | средняя | средний (на 1–2 дня) | подавать сразу 3 шаблона; пока ждём — отправка не работает, без обхода |
| Изменение полей листинга call-center (`Status`, `ClientPhone`) | низкая | высокий | строгая валидация структуры на входе, fail-fast + алерт |
| Дубль-сообщения при рестарте n8n | низкая (сделана идемпотентность) | низкий — раздражает клиента | флаги `welcome_sent_at`/`transit_sent_at`/`feedback_sent_at` в БД (см. §8.6); повторный запуск не дублирует |
| Утечка Bearer-токенов через коммит в репо | низкая | критичный | `.gitignore` для `.env`; `.env.example` без значений; pre-commit hook на `git-secrets` или `detect-secrets` |
| Падение `n8n.thefreedom.pro` | низкая | высокий | пропущенные события догоняются на возврате через diff активных; для NPS — флаг `feedback_skipped` если окно 22:00 уже прошло |
